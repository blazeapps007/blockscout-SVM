# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

See AGENTS.md for the API/indexer separate-mode architecture (APPLICATION_MODE, rules for assigning processes to `:api` vs `:indexer` mode, and cache patterns). Follow it whenever adding processes to any application's supervision tree.

## Repository Overview

Blockscout is an open-source blockchain explorer for EVM chains — an Elixir umbrella project (Elixir ~> 1.19, OTP 27). This repo is a fork of `blockscout/blockscout` (remote `upstream`). Upstream PRs target the `dev` branch with Conventional Commits titles (`feat:`, `fix:`, `chore:`, ...).

## Commands

```bash
# Setup
mix deps.get
mix compile

# Test database (required for explorer/indexer/block_scout_web tests; needs local Postgres)
MIX_ENV=test mix ecto.create
MIX_ENV=test mix ecto.migrate

# Tests — run from the individual app directory with --no-start
# (prevents supervision trees from starting; this is how CI runs them)
cd apps/explorer && mix test --no-start --exclude no_nethermind
cd apps/explorer && mix test --no-start test/explorer/chain_test.exs:42   # single test
# apps: block_scout_web, ethereum_jsonrpc, explorer, indexer, nft_media_handler, utils
# (utils has no DB/RPC deps: plain `mix test` works there)

# Mox-based RPC mocking (default; CI sets these explicitly)
export ETHEREUM_JSONRPC_CASE="EthereumJSONRPC.Case.Nethermind.Mox"
export ETHEREUM_JSONRPC_WEB_SOCKET_CASE="EthereumJSONRPC.WebSocket.Case.Mox"

# Lint / static analysis (credo and dialyzer run in MIX_ENV=test — see `cli` in mix.exs)
mix format                        # mix format --check-formatted in CI
mix credo
mix dialyzer --halt-exit-status   # slow first run (builds PLT in priv/plts)
mix sobelow --config              # run inside apps/explorer and apps/block_scout_web
npx cspell                        # spell check, config in cspell.json

# Frontend assets (apps/block_scout_web/assets)
npm install
node node_modules/webpack/bin/webpack.js --mode development   # build assets
./node_modules/.bin/eslint js/**
./node_modules/.bin/jest

# Run locally (needs Postgres + an Ethereum JSON-RPC node at localhost:8545)
mix phx.server                    # from repo root; serves at localhost:4000
# Or via Docker:
cd docker-compose && docker-compose up   # full stack incl. frontend + Rust microservices
```

`CHAIN_TYPE` (empty = default, or `ethereum`, `optimism`, `arbitrum`, `zksync`, `neon`, `celo` variant as `optimism-celo`, etc.) changes behavior and, in some modules, compiled schema code — recompile after changing it. CI runs the full test matrix across chain types (PR labels `ci:core`, `ci:all`, `ci:<chain>` select them).

## Umbrella Apps and Data Flow

```
JSON-RPC node → ethereum_jsonrpc → indexer → Explorer.Chain.import → PostgreSQL
                                                     ↑ Ecto (explorer)
                                             block_scout_web (Phoenix API/UI)
```

- **`ethereum_jsonrpc`** — JSON-RPC transport layer. Client differences (Geth, Nethermind, Erigon, Besu...) are handled via a `Variant` behaviour; transports (HTTP/IPC/WebSocket) via a callback behaviour.
- **`indexer`** — ETL pipeline of supervised fetchers (realtime + catchup). Fetchers get their `Supervisor` generated by `use Indexer.Fetcher`. Indexer does **not** touch the DB directly: it builds a map of entity lists and calls `Explorer.Chain.import/2`. Transformers in `lib/indexer/transform/` derive additional entities (addresses, token transfers...) from fetched data.
- **`explorer`** — data layer. `Explorer.Chain.import` dispatches map keys to runners in `Explorer.Chain.Import.Runner`, executed in the order defined by `Explorer.Chain.Import.Stage`. Migrations live in `apps/explorer/priv/repo/migrations`; chain-specific repos (e.g. `Explorer.Repo.Optimism`) have their own `priv/<name>/migrations`. Long-running data migrations are `Explorer.Migrator.*` modules.
- **`block_scout_web`** — Phoenix API (REST API v2 under `api/v2`, GraphQL via Absinthe) and websocket channels. On-demand fetchers (`Indexer.Fetcher.OnDemand.*`) are triggered from API requests.
- **`nft_media_handler`** — fetches/resizes/uploads NFT media; can run as a standalone worker node.
- **`utils`** — `Utils.CompileTimeEnvHelper`, `Utils.RuntimeEnvHelper`, custom Credo checks.

All runtime configuration is read from environment variables in `config/runtime.exs` with parsing helpers in `config/config_helper.exs` (`ConfigHelper`). Chain identity is `{base_type, variant}` (e.g. `{:optimism, :celo}` for `CHAIN_TYPE=optimism-celo`); `ConfigHelper.chain_type()` returns the base type.

## Key Conventions (from .github/CONTRIBUTING.md)

- **Naming**: use full words — `transaction` not `tx`, `transactions` not `txs`, `transaction_hash` not `tx_hash`, `address_hash` not `address`, `block_number` not `block_num`. API v2 fields: hashes end in `_hash` (hex strings), block numbers end in `block_number` (numbers), aggregations use plural + `_count`/`_sum`.
- **Prefer runtime configuration over compile-time.** Use `Utils.RuntimeEnvHelper` + pattern matching for feature/chain-specific function behavior. Compile-time config is acceptable only for modifying existing Ecto schemas (chain-specific fields on shared tables), and then only via `use Utils.CompileTimeEnvHelper` — never direct `Application.compile_env` (a custom Credo check enforces this).
- **New feature-specific DB tables** get their own `Ecto.Repo` (defined in `apps/explorer/lib/explorer/repo.ex`, registered in `ConfigHelper.repos/0` behind a runtime env-var check).
- **Chain/feature-specific API endpoints**: use the `chain_scope` macro in routers or the `BlockScoutWeb.Plug.CheckFeature` plug — both 404 when disabled.
- Keep `.dialyzer_ignore.exs` minimal; each suppression needs a comment explaining why it can't be fixed properly.
- Bug fixes should come as two commits: failing regression test first, then the fix.
